## Problem Statement

Korean retail investors spend significant time scanning noisy financial news to find
investment-positive signals. Existing tools (Naver Finance, brokerage research) dump
everything without filtering. There is no tool that:
1. Automatically surfaces only positive/investment-relevant news
2. Explains WHY each article is good for investors (in Korean)
3. Draws from curated, user-approved sources beyond the mainstream firehose

---

## What Makes This Cool

- **Noise filter**: only articles with `is_shown = True` appear in the frontend;
  everything gets scored and stored, but only score==1 articles surface by default
- **AI reasoning**: every score==1 article has a generated Korean "왜 좋은 뉴스인가"
  explanation — an investment thesis, not just a summary
- **Auto-pilot + manual trigger**: crawler runs on a configurable schedule OR the user
  can trigger a crawl manually via a button
- **Multi-source**: GDELT (global) + user-specified news sites (configured in a file)

---

## Constraints

- **Source approval**: adding a new crawl source requires explicit user approval.
  Copyright and terms of service must be verified before any source is added.
- **Crawl rate limits**: crawlers must respect minimum request intervals. Overly fast
  or high-volume crawling can violate site terms and carry legal risk. Each source's
  `robots.txt` and terms of service must be checked before implementation.
- **Korean-only reasons**: LLM-generated reasons are always written in Korean,
  regardless of the source article's language.
- **Language filter**: crawl Korean and English sources only (GDELT and sites alike).
- **Event Tracker out of scope**: this design covers Good News Board only.
- **Architecture docs**: existing docs are directional only — treat as incomplete
  and revise as implementation clarifies.

## Approaches Considered

### Approach A: Frontend-Only Polish
Complete the existing Next.js frontend, keep mock_db.json, no backend.
- Effort: S | Risk: Low
- Pros: Works immediately, no backend complexity
- Cons: No automation, still a manual bookmark tool, no differentiation

### Approach B: FastAPI + Multi-Source Crawler + LLM Reason Generator (CHOSEN)
Build the FastAPI backend with PostgreSQL, a configurable multi-source crawler,
and LLM scoring with auto-generated Korean reasons.
- Effort: M | Risk: Medium
- Pros: Real automation, delivers the core value (AI reason), scalable source list
- Cons: Each new source needs its own crawl logic, LLM API cost (small = minimal)

### Approach C: Full Agent Pipeline at Once
Combine all components into one agent-style workflow from day one.
- Effort: L | Risk: High
- Pros: Most automated
- Cons: Too complex, risk of building the wrong thing before validation

## Data Flow

```
[Scheduler (crawl_interval_hours in config) OR POST /api/crawl/run]
    → Crawler Engine
        → GDELT REST API query (Korean + English, finance/economy keywords)
          (note: sourcelang filter syntax TBD — validate with assignment script first)
        → User-specified sites (backend/config/crawler_sources.yaml)
            → robots.txt check before first request per site
            → minimum request interval enforced per site (rate_limit_seconds)
        → Deduplication against existing DB (by URL)
    → LLM Engine (Claude API)
        → Score each candidate article: -1 / 0 / 1
            -1: 의미없는 (no investment relevance)
             0: 적당한 (neutral / borderline)
             1: 의미있는 (clear investment-positive signal)
        → Scoring criteria from scorer_criteria.md (written before implementation)
        → Store ALL articles in DB with their score
        → If score == 1: generate Korean reason + set is_shown = True
        → If score != 1: store with is_shown = False, reason = null
        → Retry policy: exponential backoff, max 3 retries on LLM timeout/rate-limit;
          on persistent failure, skip article and log warning
    → FastAPI persists all scored articles to PostgreSQL
    → Frontend polls /api/articles (is_shown == True by default) via existing proxy
```

## User-Specified Sites (Config File)

Path: `backend/config/crawler_sources.yaml` (committed to repo)

```yaml
crawler:
  crawl_interval_hours: 24    # how often the scheduler runs automatically

sources:
  - name: "Example Finance News"
    url: "https://example.com/finance"
    enabled: true
    crawl_logic: "rss"          # rss | html_scraper | api
    rate_limit_seconds: 10      # minimum seconds between requests
    robots_txt_checked: true    # must be true before enabled: true
```

- `crawl_interval_hours` lives in the same config file as sources — one place to control all crawler behavior
- Each source requires its own crawl logic implementing a shared `CrawlerBase` interface
- `robots.txt` and terms of service must be verified; set `robots_txt_checked: true` only after
- Adding a new source requires user review and approval before merging

**Crawler interface (`CrawlerBase`):**

```python
class RawArticle:
    headline: str
    url: str
    source: str
    published_at: datetime

class CrawlerBase(ABC):
    @abstractmethod
    async def fetch(self) -> list[RawArticle]: ...
```

Concrete implementations: `RssCrawler`, `HtmlScraper`, `ApiCrawler` — each registered
by `crawl_logic` key in the config.

## Article Schema

All crawled articles are stored in the DB — including those scored -1 and 0.
This allows re-scoring if criteria change, and provides an audit log.

```python
class Article(Base):
    id: int                    # PK, auto-increment
    headline: str
    source: str                # site name
    url: str                   # unique — dedup key
    published_at: datetime     # article publish time (datetime, not date)
    collected_at: datetime     # when the crawler collected it
    score: int                 # -1 | 0 | 1 — LLM score for all articles
    reason: str | None         # Korean reason — populated only when score == 1
    is_shown: bool             # True = show in frontend (default: score==1)
    is_manual: bool            # True if user-added via form (manual override)
```

`is_shown` defaults:
- Auto-crawled: `is_shown = (score == 1)`
- Manual: `is_shown = True` always (user explicitly added it)

`score` defaults:
- Auto-crawled: LLM determines (-1 / 0 / 1)
- Manual (`is_manual = True`): `score = 1` fixed — no LLM call. User judgment supersedes LLM.

`reason` for manual articles: populated from the user's form input (the "왜 긍정적인 뉴스인가요?" field).

Users can toggle `is_shown` on any article via the edit form if they want to
override the LLM decision.

Note: both `published_at` and `collected_at` are `datetime` (not `date`).

## FastAPI Routers

| Method | Endpoint | Description |
|---|---|---|
| GET | `/api/articles` | List articles — filters: `date_from`, `date_to`, `source`, `is_shown` (default True), `page`, `limit` (default 10) |

| GET | `/health` | Liveness check |

**Auth**: write endpoints (`POST`, `PATCH`, `DELETE`, `POST /crawl/run`) are
local/trusted-network only in this phase. No external auth required. If
the service is ever exposed publicly, add a simple API key header.

## Score Definition

Scores are defined in `backend/scorer_criteria.md` (written before LLM scorer is implemented).
The three levels:

| Score | Label | Meaning | `is_shown` default | `reason` |
|---|---|---|---|---|
| 1 | 의미있는 | Clear investment-positive signal | True | Generated (Korean) |
| 0 | 적당한 | Neutral or borderline | False | null |
| -1 | 의미없는 | No investment relevance | False | null |

All three are stored in the DB. Only `is_shown = True` articles appear in the frontend.
The criteria doc defines specific examples and LLM prompt language for each level.

Manually re-added articles get `is_shown = True` and `is_manual = True` automatically.
